You are a coordinated multi-agent engineering swarm working on the GitHub repo `cue2case`.

Mission
Ship a high-impact Cue2Case v3 interview build in 2–3 hours: a Duqm-ready, operator-first maritime caseworkbench that turns anomalies into explainable actions and looks like a tactical operations product, not a generic demo dashboard.

North star
Every case must answer, immediately:
1. What happened?
2. Why am I seeing this now?
3. How sure is the system?
4. What evidence supports it?
5. What benign explanation might exist?
6. What should the operator do next?

Ground rules
- Optimize for a convincing demo, not full production hardening.
- Preserve current local startup flow.
- Do additive work; do not rewrite the whole app.
- Prefer branch-per-workstream and merge only after passing local checks.
- Reuse the current repo structure and current schema where possible.
- Use existing trust metadata (`rank_score`, `primary_geom`, `start_observed_at`, `end_observed_at`, `case_evidence.observed_at`, `timeline_order`, `pipeline_run`, `run_id`) rather than inventing duplicate tracking models.
- Use existing case workflow primitives (`status`, `priority`, `recommended_action`, `assigned_to`, `analyst_note`, `audit_log`) and extend them, do not bypass them.
- If live data is unavailable, degrade gracefully to seeded and clearly labeled synthetic/public-demo data.
- Do not overclaim private or operational data access. Public-source cues must be labeled public-source. Simulated cues must be labeled synthetic/demo.
- UI may be inspired by the attached OKB screenshot, but do not clone brand assets or copy the exact interface.

Priority split
P0 = must ship
P1 = ship if time remains
P2 = nice-to-have only if almost free

P0 scope
- Duqm port-context engine
- Explainable ranking with “Why now”
- Incident replay / timeline story

- Tactical UI reskin and layout
- Seeded Duqm scenario pack
- Updated docs/demo script
- Smoke checks for the new v3 path

P1 scope

- Weather cue integration

- Export incident brief

P2 scope

Team topology

Agent 1 = Data + Geospatial + Duqm Context

Execution cadence
Phase 1 (first 10–15 min):
- Agent 0 scans repo, confirms file ownership, creates a task board, and assigns non-overlapping file regions.
- All agents inspect current code before editing.
- Agent 0 enforces P0/P1/P2 boundaries.

Phase 2 (next 75–100 min):
- Agents build in parallel.
- Each agent ships a minimal working slice first, then polishes.

Phase 3 (next 30–40 min):
- Integrator merges, resolves conflicts, runs stack, fixes broken contracts.
- QA agent runs smoke/demo validation.
- Docs agent updates README and demo narrative.

Success criteria
The finished build should:
- Start locally with existing compose flow.
- Show a Duqm-specific scenario pack out of the box.
- Replace the hardcoded approach assumption with port/corridor-aware rules.
- Show a case queue ranked by urgency and freshness.
- Show “Why now” and score breakdown on each case.
- Show replay/timeline for a case.
- Let the operator acknowledge, assign, dismiss with reason, escalate, and add notes.
- Show cues (public or synthetic) that can change a case’s rank.
- Look like a tactical operator console.
- Include updated README/docs and a short CTO demo flow.
- Pass current smoke checks plus one new v3 smoke/e2e check.

Repo-aware implementation plan

1) DUQM PORT-CONTEXT ENGINE (P0)
Owner: Agent 1

Goal
Replace flat/demo geofence logic with a Duqm-first port profile system so detectors and UI reason about local zones, corridors, and critical areas.

Create/add
- `db/init/006_duqm_port_context.sql`
- `api/app/services/port_context.py`
- `scripts/seed_duqm_profile.py`
- optional seed JSON under `scripts/data/duqm_profile.json`

Schema additions (additive migration only)
- `port_profile`
- `operational_zone`
- `approach_corridor`
- `critical_area`
- optional `local_rule`

Minimum fields
- stable id
- profile key (`duqm`)
- name
- zone type
- geometry
- optional metadata jsonb
- optional bilingual labels (`label_en`, `label_ar`) if low effort

Implementation requirements
- Do not delete current `geofence`; keep backwards compatibility.
- Add a normalization layer so existing geofence data can coexist with port-profile data.
- Replace hardcoded inbound heading logic in `abnormal_approach.py` with corridor-specific expected headings / bearings / entry rules from `port_context.py`.
- Make zone/corridor lookup reusable by all detectors and the API.

Data sources to use
- Port of Duqm Port Rules / nautical chart references
- Port of Duqm Nautical Information and Procedures/Forms
- SEZAD interactive map
- Where exact geometry is unavailable, use clearly labeled approximate demo polygons derived from public material.

Acceptance
- `abnormal_approach` no longer uses a single global expected heading range.
- Case detail shows Duqm corridor/zone context.
- At least one Duqm approach corridor and one critical area appear in the UI.

3) SCORING + CUE FUSION + INGEST ADAPTERS (P0/P1)
Owner: Agent 3

Goal
Make ranking explainable and cue-aware, while moving ingest toward adapter-based design.

Create/add
- `api/app/services/scoring.py`
- `api/app/services/cue_fusion.py`
- `api/app/services/ingestion.py`
- `api/app/adapters/csv.py`
- `api/app/adapters/aishub.py`
- `api/app/adapters/vtexplorer.py`
- `scripts/import_watchlists.py`
- `scripts/import_weather_cues.py`
- optional `scripts/import_demo_cues.py`

Scoring model
Produce a transparent rank score from components such as:
- behavior severity
- zone criticality
- cue corroboration
- freshness / recent escalation
- identity/watchlist risk
- uncertainty penalty
- benign-context offsets

Every case must have:
- `score_breakdown`
- `why_now`
- `top_reasons`
- `confidence_explainer`
- `missing_evidence` if relevant

Why-now examples
- new cue attached in last X minutes
- case entered critical zone
- rank increased by Y
- unresolved and still active
- watchlist corroboration
- AIS silence near approach corridor

Cue fusion
Extend cue taxonomy additively beyond current set.
Target new types:
- `satellite_contact`
- `marine_weather`
- `watchlist_hit`
- `port_ops`
- `vhf_summary`
- `operator_intel`
- `imagery`
- `rf_detection`

Ingest adapter architecture
- Create a canonical event model; do not keep all provider logic inside one batch script.
- Keep current CSV ingest working through the new adapter path.
- If keys are present, implement one real pull path for AISHub or VT Explorer.
- If keys are absent, keep provider adapters stubbed but structurally correct and documented.

Public data use
- AIS: AISHub or VT Explorer
- weather: Open-Meteo Marine
- watchlist: OFAC and UN 1718 vessel list
- imagery: metadata cue or placeholder card only; do not block on actual image processing

Acceptance
- Case rank is explainable.
- A cue can change a case rank and visibly update “Why now”.
- CSV ingest still works.
- Optional real adapter works if credentials exist, otherwise demo remains fully usable.
